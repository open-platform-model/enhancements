# Risks, Drawbacks, Alternatives — `#Platform` Redesign Umbrella

## Risks and Mitigations

- **Combinatorial materialize cost.** A platform that subscribes to a wide SemVer range pulls every selected build at every `Materialize` call. A range like `>=1.0.0 <2.0.0` against a 50-build catalog adds noticeable wall time on cold caches and noticeable memory pressure on warm ones. **Mitigation:** explicit `Kernel.Materialize` is caller-driven and the natural cache boundary (D14 — kernel itself holds no cache; `library/opm/materialize/cache/` ships optional helpers that operator / CLI consumers wire up). Long-running consumers (`opm-operator/`) wire the cache keyed on the `Platform` CR's `metadata.generation` and reuse across many `Match` calls; single-shot consumers (`cli/`) opt out and rely on `cuelang.org/go/mod`'s on-disk module cache to amortise OCI pulls. If real-world ranges turn out wider than expected, a future enhancement can add a "max builds per subscription" guard or a deny-by-default pull policy.

- **Multi-version test surface explosion.** Every CUE / Go change in the umbrella has to be validated against N catalog versions instead of one. CI matrix grows with the subscription's typical breadth. **Mitigation:** the test suite pins a small representative version set (`{oldest-supported, oldest-mainline, current, current-rc}` per catalog) rather than the full materialized set. The matrix is bounded by policy, not by the registry.

- **Catalog migration burden.** The OPM core catalog at `catalog/opm/v1alpha1/` is the only catalog in scope today, but the migration pattern (drop any `#Module.#defines` wrapper, introduce `Catalog: { Version, ModulePath }`, rewire every primitive's metadata source) sets the precedent for any future third-party catalog. A botched first migration teaches everyone the wrong pattern. **Mitigation:** the catalog repackage is a discrete slice that lands with an end-to-end publish rehearsal *before* the library fixture cuts over. Documented migration steps live in the catalog repo's README.

- **OCI pull failure as a release-blocking error.** Today's CUE imports resolve at evaluation time; failures surface as CUE evaluation errors with reasonable diagnostics. After the cutover, `Materialize` is a Go-side step that can fail mid-pull with network errors, missing tags, or registry-auth issues. The release UX has a new failure mode the operator must understand. **Mitigation:** `MaterializeError` is a structured diagnostic naming the subscription `id`, the path, the filter, and the underlying OCI error. Library logs the pull plan before executing so failures are correlatable.

- **Publish-task stamping bug stamps the wrong version into a published artefact.** A bug in the `rsync` → overwrite → publish flow ships an OCI tag whose primitives carry the wrong `metadata.version`. The artefact passes `cue vet` (regex satisfied) and fails downstream at materialize-time unification — but only on the consumer that hits the bad tag. **Mitigation:** publish task asserts post-stamp that every primitive's `metadata.version` matches the requested SemVer (a `cue export` + grep is sufficient). Failure in this assertion blocks the `cue mod publish` step.

- **Consumer-pin churn from catalog-monolithic SemVer.** Per D18 the catalog is the unit of versioning — bumping `Catalog.Version` from `1.4.0` to `1.5.0` bumps every primitive's FQN even if only one schema changed. Consumers pinned to `container@1.4.0` see "alternatives: container@1.4.1, container@1.5.0" appear after every catalog publish, even though the Container schema may be byte-identical across those versions. **Mitigation:** D6's always-unify means byte-identical primitive bodies unify across SemVers without erroring (the matcher resolves the consumer's pin against the SemVer it pinned, not against an upgraded one); `#SubscriptionFilter.range` keeps multiple SemVers materialised simultaneously so the consumer's pin keeps matching until the range moves past it. Consumers re-pin on their own cadence; nothing breaks. The cost is authoring noise ("alternatives" lists grow with every catalog publish), not correctness. Per-primitive SemVer was rejected outright in D18 — see Alternatives.

- **Cross-catalog author-time vs platform-time pin drift.** D16 supports cross-catalog primitive references via standard CUE imports: a transformer in catalog A pins catalog B at A's author time (`import b "opmodel.dev/catalogs/b@v0"` plus references to `b.X.metadata.fqn`); the platform separately pins B via `#SubscriptionFilter.range` at platform-spec time. These two pins are independent surfaces and can disagree — A's transformer body may reference `…/container@1.4.0` while the platform's subscription to B excludes `1.4.0+`. The failure surfaces as `MissingFQN` / `UnifyError` at match time, which is correct but not as actionable as a structured `CrossCatalogMismatch` diagnostic would be (the operator has to read A's transformer body to learn which catalog-B SemVer it references). **Mitigation:** the same diagnostic kinds catch the failure — it doesn't hide silently. Operators subscribing to a catalog that transitively uses another catalog should mirror the same range. If real-world authoring hits this often enough to matter, a future enhancement can add a `#CatalogDependencies: [#ModulePathType]: #SubscriptionFilter` block to the manifest with kernel-side cross-checking and a `CrossCatalogMismatch` structured diagnostic — additive, no breakage. Tracked as a known follow-up to D16; not in scope for this umbrella.

- **`#ctx` cycle diagnostics are confusing for author-introduced cycles.** D1/D2's comprehension architecture makes `#ctx.components` ↔ `#names` cycles structurally impossible at the schema level — `#names` depends only on `metadata + #release`, not on `#ctx.components` (validated empirically by experiment 07-ctx-cycle-freedom). The residual concern is *author-introduced* cycles: a component whose `metadata.resourceName` is set to its own `#ctx.components.<self>.dns.fqdn`, for example, creates a cycle whose CUE diagnostic is `invalid interpolation: empty disjunction` (the `*name | #NameType` cascade is what fails first, not the cycle detector). **Mitigation:** the SPEC authoring guide should explicitly name this diagnostic form so a future debugger doesn't chase the disjunction error when the real bug is a cycle. Linting (e.g. `opm vet`) could detect the pattern statically.

- **Resource / trait / blueprint subpath typos ship silently (D19 + D21 residual surface).** Experiment 10 (`catalog-stamping-asymmetry`) empirically confirmed the deliberate asymmetry: `#Catalog.#transformers`'s pattern constraint catches transformer subpath / version drift loudly at `cue vet` time, but `#Resource` / `#Trait` / `#Blueprint` carry no equivalent schema-stamping pattern. An author who writes `metadata.modulePath: "\(id.ModulePath)/resorces/workload"` (typo) ships a primitive whose FQN is `…/resorces/workload/container@…` — `cue vet` clean, no diagnostic, drift propagates to every consumer importing the primitive. The risk is bounded (the wrong FQN cannot match against any platform-side transformer that references the canonical FQN, so the failure surfaces as a `MissingFQN` diagnostic at match time, not as a runtime correctness bug) but the diagnostic distance from cause (publish-time typo) to effect (match-time miss in a downstream consumer) is large. **Mitigations available (none in 0001's scope):** (1) publish-task lint that greps every primitive's `metadata.modulePath` for canonical-subpath conformance; (2) future `#resources` / `#traits` / `#blueprints` sibling maps on `#Catalog` with pattern-enforced subpath stamping (additive extension per D19); (3) per-catalog CI fixture asserting on exported primitive metadata. Lift the asymmetry only if real-world authoring surfaces enough typo incidents to justify the extra schema surface.

- **CI for `core/catalog.cue` cannot rely on `cue vet` to catch a bad `#transformers` pattern constraint (D19 mirror-pattern silent failure mode).** Experiment 09 (`catalog-mirror-pattern`) empirically confirmed that with `#Catalog.#transformers` hidden and no public reader, plain `cue vet` AND `cue vet -c` BOTH pass silently on a broken pattern (e.g., a future schema editor accidentally writing `metadata.modulePath` directly instead of `_md.modulePath` in the constraint body). The bug is visible only via `cue eval --all` (which traverses hidden fields) or at kernel-time materialize. **Mitigation:** core repo CI for `core/catalog.cue` runs a concrete-catalog fixture that exercises the pattern via a public reader, so the typical `cue vet` regimen catches the regression. Tracked as a graduation-criterion follow-up; the `core-schema-edit` skill should record this requirement when 0001's core/ slice lands.

- **Cold-start core-schema pull cost on first kernel operation (D24).** With the embedded `apis/core/` deleted, the first `Validate` / `Match` / `Compile` call on a freshly constructed `Kernel` triggers an OCI pull of `opmodel.dev/core@<CoreVersion>` via `cuelang.org/go/mod`. On a cold on-disk cache, this is a network round-trip plus an OCI tag resolution; on a warm cache, it is a local-disk lookup. The cost is one-time per kernel instance — subsequent operations reuse the loaded `cue.Value`. **Mitigation:** `cuelang.org/go/mod`'s on-disk module cache amortises the pull across kernel instances on the same host; long-running consumers (`opm-operator/`) pay it at most once per process; short-lived consumers (`cli/`) pay it once per invocation, mitigated by the on-disk cache. A `Kernel.Prefetch(ctx)` helper that callers can invoke immediately after construction (to surface registry-auth failures fail-fast) is a known follow-up if the lazy-first-use UX bites operators.

- **No offline / air-gapped operation by default (D24).** Today's embedded core schema means the library binary works fully offline — `Validate` / `Match` / `Compile` against in-memory CUE values needs no network. After the embed deletion, the kernel cannot operate without a reachable registry (live or mirrored). Sealed environments must pre-pull via `cuelang.org/go/mod`'s pre-pull mechanism or run against a registry mirror. **Mitigation:** the registry mirror story is the same as the catalog story — operators that air-gap their cluster already need a mirror for catalogs; the core schema rides on the same substrate. An opt-in embedded snapshot at a pinned version (`Kernel.CoreSchema embed.FS`-style override) is a documented follow-up if the air-gapped community asks for it; not in scope for 0001.

- **`Kernel.CoreVersion` is a new consumer-facing surface (D24).** Consumers must understand the field exists, what the default `"v0"` means (D12 pre-1.0 lockstep), and when to pin to a specific minor (`"v0.3.0"`). Library users who copy-paste a `Kernel{}` construction without setting `CoreVersion` get the latest `@v0` minor at pull time — usually fine, occasionally surprising when a new minor lands. **Mitigation:** library docs (README + `library/CLAUDE.md`) explicitly document the field; `cli/` and `opm-operator/` both expose pinning via their own configuration surfaces so end-users see a deliberate choice rather than an implicit default; library MIGRATIONS.md notes the field as the canonical pin for reproducible kernels.

## Drawbacks

- **Publish-time stamping requires task discipline.** Catalog authors lose the ability to `cue mod publish` directly from the source tree — they must use the stamped-build-dir task. Forgetting the task and using raw `cue mod publish` publishes the dev-default `…@0.0.0-dev` artefact. Surface drawback; mitigation is task-only publish and a CI guard that rejects publishes of `0.0.0-dev` tags.

- **Plain-CUE catalogs lose the `#Module` ergonomics.** Authors used to writing a `#Module` value with `#defines.{resources, traits, transformers}` lose the explicit wrapper. The plain-package form is simpler but unfamiliar — newcomers will write a `#Module` reflexively and discover the change only when their primitives fail to surface on a subscribed platform. Mitigation is catalog-repo docs and a deliberately broken example in the catalog README that shows the wrong-vs-right pattern side by side.

- **`#ctx` is read-only and computed; module authors cannot extend it.** The umbrella scope locks `#ctx` to the kernel-computed shape (`release` + `components` projection). Authors who want to surface their own per-component values into a typed channel have no extension point in this enhancement — that surface belongs to the future platform-capabilities enhancement (`#ctx.platform`). The open top (`...`) on `#ctx` reserves room for future additive siblings without modifying any module body. Acceptable trade-off because the alternative — open-ended `#ctx.custom` — defeats the typing.

- **The registry's CUE-time spec drifts from the kernel's runtime materialization.** A reader of the platform CUE sees subscriptions and filters; they do not see the actual set of pulled builds. Debugging "why isn't transformer X on my platform?" requires reading kernel logs or invoking a `Materialize`-then-dump command. Mitigation is making that dump command first-class (`opm platform materialize --dump`), tracked separately.

- **Backwards-compatibility is explicitly out of scope.** The existing platform fixture (`library/modules/opm_platform/`) is rewritten in lockstep; the existing catalog (`library/modules/opm/`) is repackaged to the D19 shape and republished as `opmodel.dev/catalogs/opm@0.1.0` (D23). Workspace `modules/*` modules (jellyfin, garage, …) — currently importing the legacy `opmodel.dev/opm/v1alpha1/schemas@v1` path — get rewired as a follow-on, non-blocking wave once `0.1.0` exists in the registry. Any external consumer of the current `#Platform` shape or the legacy `@v1` catalog tag breaks at the `@v0` minor bump that lands this change (D12 — `core` stays on `@v0`; the pre-1.0 rule lets breaking changes ride a minor). Acceptable because there is no external consumer today, but worth flagging as a discoverability cost for future onboarders who find the change in git history.

## Alternatives

- **Keep the Module-valued `#registry`; add a `version` field alongside `#module`.** Lets a single platform pin one version explicitly without the static-import dance. **Why not:** a single Module value is still one build at one tag; per-consumer multi-version on one platform remains impossible. Solves the wrong half of pain 1.

- **Keep MAJOR FQNs; rely on transformer predicates for version discrimination.** Authors stamp `catalog.version=1.4.0` labels on consumer Modules; transformers add the corresponding `requiredLabels`. **Why not:** opt-in by every author; version mismatches surface as generic predicate failures, not structured diagnostics; nothing structurally prevents two `1.x.x` builds from colliding in `#composedTransformers`. Defeats the goal of "exact SemVer in the FQN, distinct keys, CUE unification carries the integrity check."

- **Kernel injects `Catalog.Version` at materialize time via `FillPath`.** Skips the publish-time stamping; the catalog source stays "logically" version-free; the kernel writes the version on load. **Why not:** the OCI artefact is no longer self-consistent (FQN regex fails without a stamped version); "what version is this catalog?" becomes a runtime question rather than a property of the artefact; OCI tag and content can diverge silently. The publish-time stamping path makes drift impossible by construction.

- **Custom Go OCI client instead of `cuelang.org/go/mod`.** Decouples the kernel from CUE's module proxy; the kernel could enforce its own caching, signing, and policy. **Why not:** duplicates work CUE already does; the kernel needs nothing today that CUE's module system doesn't provide; signing / policy belong to a separate enhancement that can sit on top of either substrate. One way to publish and pull catalogs is the right shape today.

- **`#ctx.platform` lands in the umbrella alongside `#ctx.release` / `#ctx.components`.** Single-shot: ship the full typed context with the registry redesign. **Why not:** `#ctx.platform` is structurally tied to platform capabilities (`#Platform.#provides`, `#Module.#consumes`, the FQN-keyed `#Capability` model). Bundling them quadruples the design surface and forces every Open Question in the capabilities space to resolve before any of this ships. `#ctx { release, components }` plus the open top is the largest chunk that ships honestly without claiming the capabilities work is done.

- **Single-file 0001 (`#ctx`-only) + separate 0002 for the registry redesign.** Splits the umbrella into two narrower enhancements. **Why not:** the two halves share the same kernel touch (`#ModuleRelease` sets `#ctx.release` and the matcher consumes the materialized-platform output in the same code path), the same fixture rewrite (`library/modules/opm_platform/` migrates once), and the same SPEC co-update batch. Splitting forces a coordination tax that the single-PR-per-half discipline does not buy back.

- **Per-primitive SemVer.** Each primitive carries its own author-maintained `metadata.version` independent of the catalog publish version; primitives that didn't change between catalog publishes keep their existing version. **Why not:** reintroduces per-primitive authorial burden D7/D8/D9 explicitly eliminated; either the publish flow does per-primitive delta detection against the previous OCI artefact (hard) or each primitive carries a literal `metadata.version` that the publish task validates but doesn't overwrite (high authoring tax + silent-drift risk on missed bumps). With one catalog in scope and pre-1.0 status, every primitive changes frequently — per-primitive versioning would still bump nearly everything per publish. Rejected outright in D18, not deferred — leaving the door open invites a future inconsistency where some primitives have author-maintained versions and others inherit. Closing the question keeps the catalog homogeneous and reasoning simple.
