# Risks, Drawbacks, Alternatives — CLI CR Inventory, Library Kernel Adoption, and Operator Handoff

## Risks and Mitigations

- **Render parity breaks despite the shared kernel.** The whole zero-downtime claim rests on the CLI and operator computing the same render digest. If they feed the kernel different inputs — most plausibly a different materialized platform — digests diverge and handoff's no-op property is lost. D11 makes the source identical (read the same `Platform` CR spec, materialize via the same kernel calls), but a residual time-skew remains: each side materializes fresh, so a catalog version published between CLI apply and operator takeover can resolve the subscription range to a different set. **Mitigation:** D7.4's digest check *catches* any divergence before the ownership flip rather than letting it reach live resources (handoff aborts, safely); gate `accepted` on an experiment proving digest equality for the same release/Platform across both actors; consider pinning the exact materialized version-set as future hardening if the skew bites in practice.
- **Silent orphaning during Secret → CR migration.** If the CR status write fails after the Secret is deleted, the release becomes untracked and its resources orphan. **Mitigation:** D8's strict ordering — create CR and write `status.inventory` first, delete the Secret only after the status write succeeds; a mid-flight failure leaves the Secret intact and the release discoverable.
- **User flips `spec.owner` by hand, skipping handoff verification.** A `kubectl edit spec.owner: operator` bypasses D7's digest check. **Mitigation:** the operator's takeover still consults `status.inventory` for its prune set, so pruning stays correct; the worst case is a non-no-op first apply (resources patched, not deleted). Document `handoff` as the supported path; the unverified flip degrades to "maybe a blip," not "data loss."
- **CLI ships against a CUE pre-release (v0.17.0-alpha.1).** D9 forces it (MVS takes the max across `library`/`opm-operator`); an alpha can carry API churn and bugs. **Mitigation:** accepted under the single-user / no-compat stance (D14); the CLI's CUE usage is migrated to v0.17 wholesale; the risk retires when `library`/`opm-operator` move to stable v0.17. Track against the known workspace v0.16-vs-v0.17 split (`research/findings.md`).
- **Inventory-logic drift between CLI and operator breaks handoff prune parity.** If the two compute entry identity or digests differently, the operator's first post-handoff prune set could be wrong. **Mitigation (revised by D31):** the only artifact that actually crosses the actor boundary unmediated is the `InventoryEntry` wire shape written to `status.inventory.entries[]` — anchored by the CRD's OpenAPI schema, and populated from resources both actors observe identically because of kernel-render-parity (D9). The *algorithm* each actor uses to decide staleness/digest/collisions is local policy, never itself compared across actors; the one moment it would cross (the operator's first post-handoff stale-set read of CLI-written entries) is independently gated by D7.4's render-digest check, which guarantees the compared sets are equal before either side's algorithm choice matters. D13's original mitigation (share the implementation via `library`) is reverted — see D31 and OQ15 for the residual, non-safety-critical behavioral divergence (the CLI's and operator's stale-set base relations already differ) this leaves open.
- **The operator's apply path can silently force-claim a foreign or terminating object — no equivalent of the CLI's pre-apply existence check.** Both the operator's apply path (Flux's `ResourceManager.ApplyAllStaged`, always `ForceOwnership`) and the CLI's (`internal/kubernetes/apply.go`, unconditional `Force: true`) force SSA field-ownership on every apply. The CLI guards this with `PreApplyExistenceCheck` on a release's first apply — refusing if the target identity already exists foreign-owned or is mid-deletion; the operator has no equivalent at any point in its reconcile loop. A colliding apply doesn't just risk a one-time corruption: it relabels the foreign object as `opm-controller`-managed, so the existing delete-time ownership guard (`Prune`) treats it as legitimately owned from then on and cannot catch the mistake retroactively. Discovered during the D31 investigation; present-day exposure, independent of the shared-package question. **Mitigation:** none implemented today; tracked as OQ16 (`03-decisions.md`) — whether and how the operator should gain its own apply-time collision guard, designed independently rather than borrowed via shared code.
- **CLI cannot write `status` under the user's RBAC.** Status-subresource writes need `moduleinstances/status` permission the user's kubeconfig may lack (OQ3). **Mitigation:** `opm install crds` can emit an optional CLI-user Role; apply fails with a clear, actionable error (not a silent half-write) when status permission is missing.
- **Two actors apply simultaneously in the handoff window.** During the `spec.owner` flip both `opm-cli` and `opm-controller` could apply. **Mitigation:** ownership guards on both sides (D3 + `EnsureCLIMutable`) reduce it to a spec-field race that SSA resolves; the loser errors loudly. SSA on both sides (D10) is the non-negotiable enabler.
- **A1's coupled Flux/k8s.io/controller-runtime bump can silently strand downstream consumers that pin their own versions.** `k8s.io/api` v0.36.x deletes API-group packages (`scheduling/v1alpha1`, `autoscaling/v2beta2`) that `opm-operator`'s previously-pinned Flux libraries referenced, forcing a coupled bump of `cli-utils`, `pkg/apis/meta`, `pkg/runtime`, `pkg/ssa`, and `source-controller/api` alongside `k8s.io/*`/`controller-runtime` (A1, widened 2026-07-01 — see A1's own `design.md`). `opm-operator`'s own verification gates (unit/lint/e2e) only prove this bump is internally consistent — they say nothing about downstream repos that pin a specific `opm-operator` release or a specific Flux toolkit version against it. Confirmed concretely: `opm-kind-demo`'s `README.md` states it pins "Flux toolkit version, opm-operator version... No surprise upgrades" — exactly the kind of pin this bump moves underneath. Compounding this: `opm-operator`'s own e2e suite provides no live coverage of the operator-vs-real-Flux-source-controller interaction either (`test/e2e/lifecycle_test.go`/`concurrent_test.go`, the two files referencing `OCIRepository`, are pre-existing `Skip()`-stubbed TODOs) — so there is no automated signal, in either repo, that would catch a live incompatibility before a human checks. **Mitigation:** tracked as a new slice, A5 (`opm-kind-demo (+ audit other consumers)` — `downstream-flux-k8s-compat-audit`) in `planned-changes.md`, depending on A1: bump `opm-kind-demo`'s pinned versions and re-verify via a real `task kind:bootstrap`, and grep for any other repo pinning a specific `opm-operator` release or Flux toolkit version. A5 does not gate any other 0006 slice — it's downstream verification, not a prerequisite.
- **A CLI-seeded solo-cluster `Platform` is later adopted by the operator's privileged reconciler.** In a CRD-only cluster the CLI writes the singleton `cluster` Platform from its local/default source (D12, write-if-absent); when the operator is later installed it adopts that existing spec and renders *every* release against it with its privileged service account. A user who seeded that Platform thereby influences the operator's render input — a trust-on-first-use seeding path, and the one place a user-controlled Platform reaches a privileged actor. The exposure is bounded: the CLI's own render override carries **no** privilege beyond the user's RBAC (the CLI applies under the user's kubeconfig, so its blast radius equals what `kubectl apply` already allows; the operator never consumes a *local* platform, and handoff forces the cluster CR + digest check — D11/D7.4), and writing the cluster-scoped singleton already requires cluster-scoped `Platform` write permission, itself a privileged operation. The render layer is also hermetic — transformers are CUE, evaluated without arbitrary host code execution — so an attacker-controlled registry yields attacker-chosen *manifests* (bounded again by apply-time RBAC), not host RCE. **Mitigation:** operator install should surface and validate the existing `Platform` it is about to adopt rather than blindly trusting it; pin the write-if-absent mechanism so a concurrent operator install cannot clobber or be clobbered (OQ13). The asymmetry is deliberate — the apply path needs no gating (D17/D21), only the operator-adoption path does.

## Drawbacks

- **CRDs become a hard prerequisite for every CLI apply.** Today the CLI works on any cluster with no setup; after this, `opm install crds` is a required first step even for users who never want the operator. Cost: a higher barrier to the very first `apply`. Accepted because CR-backed inventory is what makes handoff possible, and the install is one offline-capable command.
- **The CLI takes a dependency on `library` (and the CUE alpha it carries).** From its own render path to importing the kernel + shared inventory package, pulling in the kernel's transitive closure and CUE v0.17.0-alpha.1. D13 keeps it to `library` *only* — no controller-runtime, no Flux, no `opm-operator` module edge — so the cost is one dependency and one render/match + inventory implementation. Accepted.
- **Duplicated/ported reconcile + prune logic (D13).** The CLI's one-shot apply/prune ports the operator's reconcile concepts rather than importing them. Cost: a behavioural-parity surface (phase order, ownership guard, safety exclusions) to keep aligned by review. Accepted because importing the operator's controller-runtime-based machinery into a one-shot CLI is the wrong shape; the *pure* parity-critical logic (entry identity/digest) is shared via `library`, so only the orchestration is ported.
- **The kernel-adoption strand is gated on 0001.** D9 cannot fully land until 0001's `library` slice ships, so 0006's timeline is partly bound to 0001's. Accepted, with OQ5 offering a two-wave split so the inventory/handoff value need not wait.
- **Duplicated apply logic (D10).** The CLI and operator maintain separate SSA apply paths that must stay behaviourally close. Cost: a real divergence surface (staging order, conflict handling). Accepted because unifying the apply engine is a large separable effort and digest verification + SSA already make handoff safe without it.

## Alternatives

- **Share only the inventory store, keep the CLI's own render pipeline.** Closes the disjoint-store gap but not the render-divergence gap; handoff digest parity stays best-effort. **Why not:** half the convergence delivers none of the zero-downtime safety — see D9.
- **Unify the apply engine too (one shared apply package for both actors).** Maximal convergence. **Why not:** lifting the operator's Flux/controller-runtime apply into a CLI-usable package is a large, separable effort that handoff does not require (D7.4 + SSA suffice) — deferred, not rejected.
- **Import `opm-operator/api/v1alpha1` + `pkg/inventory` for the typed CR and inventory (RFC-0007 D4).** **Why not:** drags controller-runtime + Flux + apiextensions into a one-shot CLI (package-granular imports — `research/findings.md`). Chosen instead (D13): pure inventory logic in `library`, CR as `unstructured`, no `opm-operator` module edge. The CRD types stay Kubebuilder-generated in `opm-operator` as the serialization shape.
- **A separate CLI-only inventory CRD.** Avoids importing the operator's type. **Why not:** a distinct type defeats the entire handoff goal — both actors must read and write the same object.
